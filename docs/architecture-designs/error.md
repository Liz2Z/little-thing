# error design


这是一个非常经典且在软件架构设计中经常被讨论的问题。简单来说，**是否有必要取决于你的系统规模、团队协作模式以及系统的对外交互复杂度。** 

对于小型项目，这可能是过度设计（Over-engineering）；但对于中大型项目、微服务架构或提供公共 API 的系统，**统一的错误创建收口（如 `createError`）和全局统一的错误码（Error Code）是非常有必要甚至是必不可少的。**

我们可以从“统一错误创建”和“统一错误码”两个维度来深入剖析它的利弊及最佳实践：

---

### 一、 统一错误创建（如收口到 `createError` 或基类工厂）

将错误对象的实例化收口到一个工厂函数（`createError`）或一个统一的基类（如 `BaseError`），有以下核心影响：

#### **优势（为什么需要它？）**
1. **结构一致性**：保证系统中抛出的所有错误都具备相同的结构（例如都包含 `code`, `message`, `httpStatus`, `details`, `traceId` 等）。这让中间件（如 Express/Koa/Spring 的全局错误拦截器）处理起来极其简单，不需要写大量的 `if-else` 去判断错误类型。
2. **全局植入能力**：如果你想在所有错误中自动注入当前请求的 `traceId`（链路追踪 ID），或者在创建错误时自动触发 metrics 打点计数，收口到 `createError` 让你只需要改一行代码。
3. **防止内部信息泄露**：在 `createError` 中可以强制区分“面向开发者的内部信息（internalMessage）”和“面向用户的安全提示（publicMessage）”，避免把数据库 SQL 报错直接丢给前端。

#### **劣势与陷阱**
1. **调用栈（Stack Trace）污染**：如果只是简单封装一个函数，错误抛出时的调用栈顶部可能会永远是指向 `createError` 这个函数，而不是真正出问题的业务代码位置。需要在代码中特别处理（比如 JS/Node.js 中使用 `Error.captureStackTrace` 去除工厂函数的栈帧）。
2. **缺乏类型语义**：如果系统里所有错误都是用 `createError({ code: 1001, message: '...' })` 抛出，代码阅读者很难直观看出这是一个“权限错误”还是“数据库错误”。

---

### 二、 统一的错误码（Error Code）

错误码（如 `USER_1001`, `ERR_INVALID_PARAM`）是系统之间、前后端之间沟通的“契约”。

#### **优势（为什么需要它？）**
1. **对客户端友好与多端一致性**：前端/客户端绝对不应该通过判断 `message` 的文本内容来做业务逻辑分支（因为文案随时会变，且有国际化需求）。前端应该只认 Error Code。
2. **国际化（I18n）的基础**：后端只需抛出 `code: 'AUTH_001'`，前端或网关层根据当前用户的语言环境，将 `AUTH_001` 映射为“密码错误”或“Incorrect password”。
3. **监控告警与日志分析**：在 Kibana/Grafana/Sentry 中，通过 Error Code 聚合错误日志，能极快地发现系统的主要故障点（比如突发大量 `PAYMENT_502` 错误码，立刻触发报警）。
4. **提升文档质量**：API 文档中可以清晰地列出每个接口可能返回的错误码列表，前端或对接的第三方能清晰地知道需要处理哪些异常。

#### **劣势与陷阱**
1. **维护成本高（中心化噩梦）**：如果把所有的 Error Code 写在一个几十万行的全局枚举文件里，每次多人开发合并代码时都会产生冲突。
2. **心智负担**：开发者在抛出错误前，需要去查阅“这个错误该用哪个码”，找不到时还得去申请新建错误码。

---

### 三、 行业最佳实践方案讨论

为了吸收上述两者的优点并规避缺点，现代软件工程通常推荐以下**“折中且灵活”的架构方案**：

#### 1. 错误码按领域（Domain）划分，避免大一统的“上帝文件”
不要试图用一个全局的 `ErrorCode.ts` 管理所有模块。推荐采用**前缀+模块+编号**的方式，按模块分散管理。
* 例如：`AUTH-1001`（认证模块参数错误），`ORDER-2005`（订单库存不足）。
* 每个微服务或每个业务模块维护自己的 Error Code 枚举文件。

#### 2. 使用“分类基类 + 工厂方法”，而不是单一的 `createError`
比起一个包揽所有的 `createError` 函数，更好的做法是定义具有语义的 Error 子类或特定领域的创建函数。
例如，在 TypeScript/Node.js 中：
```typescript
// 好的做法：语义化工厂或类
throw new UnauthorizedError(ErrorCode.AUTH_TOKEN_EXPIRED);
throw new BusinessError(ErrorCode.ORDER_OUT_OF_STOCK, { itemId: 123 });

// 较差的做法：纯配置的通用函数
throw createError(401, 10002, "Token expired");
```
这样做既保留了统一基类（`UnauthorizedError` 继承自 `BaseError`）的一致性，又让业务代码具有高度的**可读性**。

#### 3. 分层错误处理（核心层抛出异常，网关/接口层映射错误码）
* **在 Domain / 业务逻辑层**：尽量抛出语义化异常（Exception），不需要太关心 HTTP 状态码等网络细节。
* **在 Controller / 全局拦截器层**：捕获这些异常，然后通过一个统一的 `ErrorMapper`（错误映射器），将异常转换为包含统一 Error Code 和 HTTP Status 的标准响应格式（如符合 RFC 7807 Problem Details 标准的格式），统一返回给前端。

### 总结建议

如果你现在正在做技术选型或架构重构：
* **必须做**：前后端交互的接口，**必须**有一套统一格式的错误数据结构（包含 Error Code）。
* **应该做**：在代码层面，将所有自定义业务错误的基类统一收口（例如都继承自 `AppError`），以保证日志格式一致和拦截器处理方便。
* **避免做**：不要把所有的错误码维护在一个巨型文件里；不要为了用 `createError` 而牺牲代码的语义化和真实的调用栈信息。
