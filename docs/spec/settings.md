## 问题陈述

现有 Settings 配置系统设计存在以下问题：

1. **Config 和 Settings 概念混淆**：原有设计没有明确区分技术配置和用户偏好
2. **Provider 配置设计不当**：Schema 设计与实际代码实现脱节，代码中直接从环境变量读取
3. **缺少多 Provider 支持**：无法支持会话级别的 Provider 选择
4. **层级加载不完整**：没有实现完整的 4 层级加载机制

## 需求证据

从用户访谈中收集到的具体需求：

- "需要会话级别的 provider" —— 用户明确要求每个会话可以选择不同的 Provider
- "多个独立 Provider" —— 需要同时配置 Anthropic、OpenAI、智谱、Moonshot 等多个 Provider
- "智谱作为默认" —— 明确指定默认 Provider 为智谱 AI
- "配置了 base url 和 key 后，是不是可以直接能拉取到模型列表？" —— 表达了对动态模型列表的需求
- "内置默认列表，运行时支持增加自定义个 provider" —— 明确的混合模式偏好

## 现状分析

**现有代码问题**：
```typescript
// packages/server/src/providers/anthropic.ts
constructor() {
  const llmConfig = {
    apiKey: process.env.LLM_API_KEY || '',
    baseUrl: `${process.env.LLM_BASE_URL}/v1`,
    model: process.env.LLM_MODEL || 'glm-4.7',
  };
  this.config = llmConfig;
}
```

这段代码完全绕过了设计文档中的 Schema，直接读取环境变量，导致设计文档与实现脱节。

**用户实际需求**：
- 支持多个独立 Provider，按会话选择
- 运行时可添加自定义 Provider
- 配置与敏感信息分离存储

## 约束条件

1. **单用户系统**：不需要多用户隔离和认证机制
2. **配置格式**：使用 JSON 格式
3. **文件位置**：遵循 XDG 规范，`~/.config/littlething/`
4. **安全要求**：敏感信息（API Keys）独立存储在 credentials 文件中

## 前提假设

1. **单用户系统**：Settings 是单用户系统的配置中心，不需要多用户隔离和认证机制

2. **Provider 配置**：Provider 配置包括内置和自定义两类，内置 Provider 有预定义的默认模型列表

3. **credentials 独立**：credentials 文件只包含敏感信息（API Keys），不包含其他配置

4. **会话级 Provider 选择**：Settings 中配置多个可用 Provider，有一个默认 Provider。创建会话时可以选择使用哪个 Provider，如果未指定则使用默认 Provider

5. **UI 预留入口**：UI 偏好配置目前只预留入口，具体字段后续定义

## 方案对比

### 方案 A：简洁扁平型（未采纳）

**特点**：结构最简洁，字段嵌套最少

**优点**：
- 易于理解
- 访问路径短

**缺点**：
- 模块边界不清晰
- 扩展时容易混乱

### 方案 B：模块分离型（推荐）✅

**特点**：按功能模块分类，结构清晰

**优点**：
- 模块边界清晰
- 扩展性好
- 符合单一职责原则
- 易于测试和维护

**缺点**：
- 嵌套层级稍深

**结构**：
```typescript
{
  llm: { defaultProvider, providers, customProviders },
  server: { port, host },
  logging: { level, format },
  ui: {}
}
```

### 方案 C：分层优先级型（未采纳）

**特点**：默认值与具体值分离

**优点**：
- 配置复用性好
- 减少冗余

**缺点**：
- 结构复杂
- 理解成本高
- 合并逻辑复杂

## 推荐方案

**选择方案 B（模块分离型）**

理由：
1. **清晰性**：每个模块职责明确，llm/server/logging/ui 各司其职
2. **扩展性**：添加新配置项（如存储配置）只需添加新模块
3. **可维护性**：模块化结构便于单元测试和代码定位
4. **符合直觉**：开发者能够快速找到相关配置

## 层级加载策略

**配置文件层级**（优先级从高到低）：

1. **命令行参数**（最高优先级）
2. **环境变量直接覆盖**
3. **项目配置**：当前项目目录下的 `.littlething/settings.json`
4. **全局配置**：`~/.config/littlething/settings.json`（最低优先级）

**文件位置**：
- 全局配置：`~/.config/littlething/settings.json`
- 项目配置：`{项目目录}/.littlething/settings.json`
- 敏感信息：`~/.config/littlething/credentials`

## Zod Schema

```typescript
import { z } from 'zod';
// LLM 配置 Schema
const llmSchema = z.object({
  provider: z.string().default('zhipu'),
  model: z.string().default('glm-4.7'),
  baseUrl: z.string().url().optional(),
  timeout: z.number().min(1000).max(120000).default(30000),
  maxRetries: z.number().min(0).max(5).default(3),
});

const providerSchema = z.object({
  models: z.array(z.string()).optional(),
  baseUrl: z.string().url(),
  timeout: z.number().min(1000).max(120000).default(30000),
  maxRetries: z.number().min(0).max(5).default(3),
});

const providersSchema = z.record(providerSchema);

// 服务器配置 Schema
const serverSchema = z.object({
  port: z.number().min(1).max(65535).default(3000),
  host: z.string().default('localhost'),
});

// 日志配置 Schema
const loggingSchema = z.object({
  level: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
  format: z.enum(['text', 'json']).default('text'),
});

// 完整 Settings Schema
export const settingsSchema = z.object({
  llm: llmSchema,
  server: serverSchema,
  logging: loggingSchema,
  ui: z.record(z.unknown()).default({}),
});

export type Settings = z.infer<typeof settingsSchema>;
```

## Provider 验证策略

### 添加自定义 Provider 时的验证

```typescript
// 添加自定义 Provider（运行时，不持久化）
export function addCustomProvider(name: string, config: ProviderConfig): void;
```

**验证时机**：立即验证

**验证内容**：
1. **Schema 验证**：使用 Zod 验证配置结构是否正确
2. **URL 格式验证**：验证 baseUrl 是否是有效的 URL
3. **名称冲突检查**：检查是否与内置 Provider 或其他自定义 Provider 冲突
4. **可达性验证**：**可选**，是否验证 baseUrl 可达性取决于 `validateUrl` 参数

**验证级别**：
```typescript
interface AddProviderOptions {
  validateUrl?: boolean;  // 是否验证 URL 可达性，默认: false
  checkCredentials?: boolean;  // 是否检查 credentials 中存在对应的 key，默认: true
}

export function addCustomProvider(
  name: string,
  config: ProviderConfig,
  options?: AddProviderOptions
): void;
```

### 启动时的 Provider 验证

**验证时机**：配置加载后

**验证内容**：
1. **defaultProvider 存在性**：验证 defaultProvider 指向的 Provider 是否存在
2. **credentials 完整性**：警告（非错误）如果某个 Provider 没有 credentials
3. **配置完整性**：每个 Provider 至少要有 baseUrl

**验证策略**：默认 `warn` 模式，允许启动但有警告

## 会话配置 Schema

```typescript
// 会话级 Provider 选择
interface SessionConfig {
  id: string;
  name: string;
  provider: string;     // Provider 名称（必须在 settings.llm.providers 或 customProviders 中）
  model: string;        // 模型名称（从 Provider API 获取的模型列表中选择）
  createdAt: string;    // ISO 8601 格式
  updatedAt: string;
}

// 创建会话选项
interface CreateSessionOptions {
  name?: string;
  provider?: string;    // 未指定时使用 settings.llm.defaultProvider
  model?: string;       // 未指定时使用 Provider 的第一个可用模型（按字母序排序）
}
```

### 会话与 Provider 生命周期的关联

**创建会话时**：
1. 验证指定的 Provider 是否存在于配置中
2. 验证指定的模型是否在该 Provider 的可用模型列表中
3. 保存 Provider 名称和模型名称到会话配置

**使用会话时**：
1. 读取会话配置中的 Provider 名称
2. 检查该 Provider 是否仍然存在
3. 如果 Provider 不存在：
   - 抛出错误并提供修复建议（重新配置 Provider 或选择其他 Provider）
   - 或提供迁移工具将历史会话迁移到新的 Provider

**Provider 配置变更时**：
1. 已创建的会话不受影响（会话保存的是快照）
2. 新会话使用更新后的配置
3. 提供工具批量更新历史会话的 Provider

**模型名称变更时**：
1. 如果模型不再可用，会话加载时提示用户重新选择模型
2. 提供工具批量更新历史会话的模型名称

## 安全最佳实践

### credentials 文件管理

**文件权限**：
- credentials 文件应设置为 `600`（仅所有者可读写）
- 在创建文件时自动设置正确权限
- 启动时检查文件权限，如不安全则警告

**示例代码**：
```typescript
import { chmod, existsSync } from 'fs';
import { writeFile } from 'fs/promises';

const CREDENTIALS_PATH = '~/.config/littlething/credentials';
const RECOMMENDED_PERMISSIONS = 0o600;

export async function ensureCredentialsPermissions(): Promise<void> {
  if (existsSync(CREDENTIALS_PATH)) {
    await chmod(CREDENTIALS_PATH, RECOMMENDED_PERMISSIONS);
  }
}

export async function saveCredentials(credentials: Credentials): Promise<void> {
  await writeFile(CREDENTIALS_PATH, JSON.stringify(credentials, null, 2), { mode: RECOMMENDED_PERMISSIONS });
}
```

**密钥轮换**：
- 提供命令更新 credentials 中的 API Key
- 更新后清除相关缓存
- 不记录日志或打印到控制台

**密钥验证**：
- 启动时验证 API Key 格式（如长度、前缀）
- 调用 API 前验证 Key 是否存在
- API 调用失败时提示检查配置

### 敏感信息处理

**不要记录敏感信息**：
- API Key 不应出现在日志中
- 错误消息中不暴露完整的 API Key
- 调试信息中用 `***` 替换敏感部分

**示例**：
```typescript
// 错误的日志记录
console.log(`Using API Key: ${apiKey}`);  // ❌

// 正确的日志记录
console.log(`Using API Key: ${apiKey.slice(0, 4)}...${apiKey.slice(-4)}`);  // ✅
console.log(`Using API Key: ***`);  // ✅
```

## 错误处理策略

### 配置加载错误

根据 `errorHandling` 选项，系统有三种错误处理模式：

**strict（默认）**：
- 任何配置错误都会导致启动失败
- 抛出具体的错误信息，指导用户修复
- 适用于生产环境

**warn**：
- 配置错误时输出警告但继续运行
- 使用默认值替代错误配置
- 适用于开发环境

**fallback**：
- 配置错误时静默降级到默认值
- 不输出任何警告
- 适用于测试环境

### 常见错误场景

| 错误场景 | strict | warn | fallback |
|----------|--------|------|----------|
| defaultProvider 指向不存在的 Provider | 启动失败，报错 | 警告，使用第一个可用 Provider | 静默使用第一个可用 Provider |
| credentials 文件缺失 | 启动失败，报错 | 警告，API Key 为 undefined | 静默，API Key 为 undefined |
| 配置文件 JSON 格式错误 | 启动失败，报错 | 警告，忽略该配置文件 | 静默忽略该配置文件 |
| baseUrl 不是有效的 URL | 启动失败，报错 | 警告，跳过该 Provider | 静默跳过该 Provider |

## 环境变量处理

### 支持的语法

credentials 文件支持以下环境变量引用语法：

```json
{
  "providers": {
    // 必需的环境变量，不存在时报错
    "zhipu": "${ZHIPU_API_KEY}",

    // 可选的环境变量，不存在时使用默认值
    "anthropic": "${ANTHROPIC_API_KEY:-sk-default-key}",

    // 普通字符串（无环境变量引用）
    "custom": "hardcoded-key"
  }
}
```

### 解析规则

1. **`${VAR}`**：必需的环境变量，不存在时抛出错误
2. **`${VAR:-default}`**：可选的环境变量，不存在时使用默认值
3. **`$$`**：转义，输出字面量 `$`
4. **不支持的语法**：嵌套引用、条件值、命令替换

### 实现示例

```typescript
const ENV_VAR_PATTERN = /\$\{([^}:]+)(?::-([^}]*))?\}/g;

export function expandEnvVar(value: string): string {
  return value.replace(ENV_VAR_PATTERN, (_, varName, defaultValue) => {
    if (process.env[varName] !== undefined) {
      return process.env[varName]!;
    }
    if (defaultValue !== undefined) {
      return defaultValue;
    }
    throw new CredentialsError(`Environment variable not found: ${varName}`);
  });
}
```

### 向后兼容性

为保持向后兼容，环境变量优先级高于配置文件：

```
命令行参数 > 环境变量 > 项目配置 > 全局配置
```

现有的环境变量（`LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL`）仍然有效，但推荐迁移到新的配置系统。
